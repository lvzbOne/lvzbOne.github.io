使用git的最小化配置

使用git时，需进行最小化配置 用户和email信息，常见案例如下。git config –list 列出配置环境

类似命令：git config –global , git config –local, git config –system.

  • git config –local ##只对某个仓库有效,切换到另外一个仓库失效
  • git config –global ##当前用户的所有仓库有效,工作当中最常用
  • git config –system ##系统的所有用户,几乎不用

local的在.git/config里面；global的在个人home目录下的.gitconfig里面；system应该在git安装目录的下

PS C:\Users\lvzb3> git config --global user.name 'lvzb31988'
PS C:\Users\lvzb3> git config --global user.email 'lvzb31988@163.com'
PS C:\Users\lvzb3> git config --global --list
user.name=lvzb31988
user.email=lvzb31988@163.com

认识工作区暂存区和本地仓库

  1. 工作区就是指我们当前的操作环境
  2. 暂存区就是，当我们在当前环境更改了后，想暂时保存到某个位置时但是又不想正式提交，就可以通过git add XXX 把想要暂存的内容提交到暂存区里，以便我们可以后续需要的时候从暂存区里取出来
  3. git本地仓库，则是当我们感觉这个文件可以正式提交时，就可以把文件从暂存区提交到git本地仓库，在git里文件一定是被先提交到暂存区里后，才能被提交到本地仓库，否则就会报错提示。

  • git add -u：将文件的修改、文件的删除，添加到暂存区。

  • git add . ：将文件的修改，文件的新建，添加到暂存区。

  • git status : 查看当前git 仓库下的文件状态？

  • git commit -m ‘本次提交的备注和描述内容’ ： 提交暂存区的文件到本地仓库

本地分支和远程分支

  • 基于远程分支创建本地分支: git checkout -b local/develop origin/develop,最好创建同名的分支这样就能直接使用git push
  • 本地分支直接推送到远程分支: git push -u origin b-test:test 这样会讲本地分支推送远程分支并且创建远程分支 test
  • 本地分支推送到远程分支: git push origin local/develop:develop
  • 本地分支和远程分支进行关联（绑定）：
  • 查看本地分支和远程分支关系: git branch -vv
  • 切换分支： git checkout 分支名称
  • 分支合并：先切换到某个commit少的分支 然后执行 git merge local/feature，处理冲突然后git add,git commit，git push 即可
  • 分支删除：git branch -d <branch-name>, git branch -D <branch-name> 大D是强制删除，d是分支的文件都提交了才能删除 , 删除远程分支：git push origin --delete local/feature

是的，你可以通过使用 git push 命令将本地分支推送到远程分支，并自动在远程仓库创建该分支。以下是操作步骤：

确保你已经完成了在本地创建分支并进行了相关的提交操作。
使用以下命令将本地分支推送到远程分支，并在远程仓库创建该分支：
 
git push -u origin local/feature:feature

上述命令中，-u 参数用于设置本地分支与远程分支的关联关系，以便之后的 git push 命令可以自动将本地分支推送到远程分支。

通过执行上述命令，Git 将会把本地分支 local/feature 推送到远程分支 feature，如果远程分支不存在，Git 会自动在远程仓库创建该分支。

请确保你有足够的权限进行推送操作，并且远程仓库允许创建分支的操作。

创建git的 tag

简而言之，git tag能帮我们打上一个标签，通过该标签我们能下载当时commit的代码，并且我们也可以通过tag在gitee或者github里面创建release 版本，甚至上传响应的源码包

  • tag 是 Git 版本库的一个快照，指向某个 commit 的指针
  • tag 的存在，是因为我们需要这种标记的功能。目前的项目开发中，当发布版本时 tag 就派上用场了。例如 v1.0.1，v1.0.2… 另外，git 提供了 tag 的增删改查一系列操作，在 tag 的使用上，可谓非常之方便。
  • tag 和 branch 有点相似，但是它们的职责分工和本质都是不同的。
    • tag 对应某次 commit, 是一个点，是不可移动的
    • branch 对应一系列 commit，是很多点连成的一根线，有一个HEAD 指针，是可以依靠 HEAD 指针移动的。
    • 改动代码用 branch ,不改动只查看用 tag。
    • tag 和 branch 的相互配合使用，有时候起到非常方便的效果，例如 已经发布了 v1.0 v2.0 v3.0 三个版本，这个时候，我突然想不改现有代码的前提下，在 v2.0 的基础上加个新功能，作为 v4.0 发布。就可以 检出 v2.0 的代码作为一个 branch ，然后作为开发分支。
  • 注意tag 创建后，push到远程仓库需要额外的push tag
# 创建 一个tag XXX 为commit的hash，不带-a时默认是最新的当前分支
git tag -a v0.5 xxxxxx
git tag 

## 删除本地tag
git tag -d v0.5

## 推送本地tag到远程仓库
git push origin tag v1.0

## 删除远程仓库的tag
git push origin :<oldTagName>
